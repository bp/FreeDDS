When piping a dataset form one application to another, by default, the binary is attached to the Dictionary so that they can both be passed from "stdout:" of one process to "stdin:" of another process. When piping to a non-DDS program that expects only a Standard Format binary (and no DDS Dictionary) then use "out_data=stdout:" to cause DDS to pass only the binary (see I/O Control). In addition, when passing USP binary data thru a pipe, DDS keeps the data in host-dependent formats (eg. Little Endian on Linux) to be compatible with USP programs.

Two restrictions arise when piping data:

  1. Piped data cannot be randomly accessed. A few applications may need to perform random I/O and thus will not work for piped data. For example, many migration codes accumulate the results into the output data, thus reading and writing traces multiple times.

  2. Piped data cannot be accessed by MPI codes. However, if only one MPI node accesses data, then a temporary "named pipe" can be setup (see man page for mknod; eg. "mknod pipename p"). One process can write to the named pipe and the other can read from it just so long there is no seeks done and only one node is accessing the pipe.
