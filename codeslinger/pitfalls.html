Trying too hard

DDS can be awkward. Especially when using older, distributed applications as templates. Many older programs were written before the advent of the convenience routines and therefore use the more flexible, but mind-numbing, double buffering technique. This always obscures the intended purpose of the application.

When practical, start with templates in the Basic Example Codes area.

Bad format strings passed to fdds_scanf and fdds_printf

The 'fmt' arguments used for the fdds_scanf, fdds_printf, fdds_prtcon and friends MUST use the NULL terminator '\0'. Fortran-to-C character string hand-offs are notoriously troublesome. The NULL terminator is a stable solution to this problem.

Too few arguments in varargs area of *dds_printf

The number of arguments converted by the *dds_printf, *dds_prtcon, *dds_prtmsg, and *dds_prterr functions will exactly match the format conversion requests in the 'fmt' argument. If you pass fewer variables than the conversion requests, the program will nearly always die.

Scanning for parameters from the wrong dictionary

It's easy to forget which dictionary is in 'scan' mode. Remember that after opening an input dataset, the input dictionary is set for scanning and when opening an output dataset, the output dictionary is set for printing. However, when interleaving scans of data dictionaries with scans of the par: dictionary, it may save headaches to precede each block of scans with *dds_dict or *ddx_dict, as appropriate.

Wrong file permissions for input

Ocasionally, failure to have write permissions to a file has been known to cause read failures. This is known to happen when copying read-only data from a DVD to a hard disk. (A fix to this was reported May, 2009)
