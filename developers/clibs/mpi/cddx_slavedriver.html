cddx_slavedriver - master-slave parallelization scheme using DDS and MPI

SYNOPSIS

#include <pcddx.h>

int cddx_slavedriver(np, node, numjobs, verbose, mydata, Slave_Work, Slave_SendResult, Slave_SendFinalResults, Master_ReceiveResult, Master_ReceiveFinalResults, Master_BroadcastResults)

int np
int node
int numjobs
int verbose
void mydata(*)
void (*Slave_Work)( int *jobnumber, void *mydata )
void (*Slave_SendResult)( int *np, int *node, int *jobnumber, void *mydata )
void (*Slave_SendFinalResults)( void *mydata )
void (*Master_ReceiveResult)( int *np, int *slave, int *jobnumber, void *mydata )
void (*Master_ReceiveFinalResults)( void *mydata )
void (*Master_BroadcastResults)( void *mydata )

DESCRIPTION

cddx_slavedriver provides a template for a master-slave parallelization. Parallel programming is non-trivial. The goal of this routines is to provide a framework for the developer, so that they may concentrate on the algorithm, rather than the parallelism. That said, thought should be given to the level of parallelism required in any application. If the level of parallelism is too fine (the jobs are too short) then there is a potential for a bottleneck with the inter-process communication. If the level parallelism is too coarse (the jobs are too long), then you may not be able to use many threads.

The runtime parameter masterslave_debug= can control the behaviour of this routine for debugging purposes.
Valid values are:

  • 0: normal mode, with no debugging (this it the default value).
  • 1: the master will stop after giving one job out.
  • 2: the master will stop after giving one job out to each slave.

INPUT

The parameter, "np", specifies the number of MPI threads used by the program.

The parameter, "node", specifies the number of the current MPI thread.

The parameter, "numjobs", specifies the total number of jobs to be done.

The parameter, "verbose", specifies whether to print fewer (verbose=0) or more messages (verbose=1).

The parameter, "mydata", specifies a work array (or just a pointer if you are mixing Fortran and C) of data that can be used by the used-defined functions that follow.

The parameter, "Slave_Work", defines a subroutine that will be called by the slaves to do a single element of work.

The parameter, "Slave_SendResult", defines a subroutine that will be called by the slaves to send back results of a single element of work.
Note that if the slaves are keeping all of the results until the end, then this routine may be empty, but it must exist.

The parameter, "Slave_SendFinalResults", defines a subroutine that will be called by the slaves, after they have finished all the jobs, to send back any final results.
Note that if the slaves are sending back individual results after completing each job, then this routine may be empty, but it must exist.

The parameter, "Master_ReceiveResult", defines a subroutine that will be called by the master to receive results of a single element of work.
Note that if the slaves are keeping all of the results until the end, then this routine may be empty, but it must exist.

The parameter, "Master_ReceiveFinalResults", defines a subroutine that will be called by the master, after the slaves have finished all the jobs, to receive any final results.
Note that if the slaves are sending back individual results after completing each job, then this routine may be empty, but it must exist.

The parameter, "Master_BroadcastResults", defines a subroutine that will be called by the master AND the slaves, after the slaves have finished all the jobs, to receive any final results.
Note that this routine may be empty, but it must exist.

RETURN VALUE

cddx_slavedriver always returns 0.
